0Architecture & transport

app.ts is the router in a thin-client design: stateful <pi-*> custom elements own their local DOM, while app.ts translates their events into HTTP POSTs and translates a one-way Server-Sent Events stream back into DOM mutations. There is no shared client/server state object — the SSE frame stream is the state.

1Bootstrap & module wiring

The custom elements are pulled in as bare side-effect imports. This is load-bearing — the header comment warns that a type-only import gets elided and tree-shaken, dropping the customElements.define() call so createElement("pi-tool") silently returns an inert element.

app.ts — imports

import "./pi-frame.ts";   // side-effect: registers <pi-frame>
import "./pi-tool.ts";
import "./pi-thinking.ts";
import type { PiTool } from "./pi-tool.ts"; // names for annotations only

The registry is exposed on window.piweb so client extensions can override tool rendering, and the built-in edit→diff renderer is registered immediately:

2POST helpers & thread routing

Every mutation flows through two helpers. postThread stamps the request with activeThreadId so the host routes it to the correct AgentSession; post is used when a different thread is the target (e.g. deleting a background thread).

app.ts — post / postThread

function post(path: string, body: any) {
    return fetch(path, {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify(body),
    });
}
function postThread(path: string, body: any = {}) {
    return post(path, { ...body, threadId: activeThreadId });
}
Source of truth: activeThreadId always mirrors the URL query ?thread=<id>. That makes the active thread survive refresh, be bookmarkable, and lets separate tabs view separate threads. urlThread() reads it back from location.href.

3Composer → runInput dispatch

The <pi-composer> element emits three events; app.ts binds them:

composer.addEventListener("pi-dequeue", () => restoreQueue({ abort: false }));

runInput() (line 1713) is the single dispatch point that decides prompt vs. shell vs. command:

app.ts:1713 — runInput

function runInput(text: string, images: any[] = []) {
    text = (text ?? "").trim();
    if (!text && !images.length) return;

    if (text.startsWith("!")) {                       // ── shell ──
        const exclude = text.startsWith("!!");      // !! = keep out of context
        const command = text.slice(exclude ? 2 : 1).trim();
        if (command) postThread("/bash", { command, excludeFromContext: exclude });
        return;
    }
    if (text.startsWith("/")) {                       // ── command ──
        const sp = text.indexOf(" ");
        const cmd = sp === -1 ? text : text.slice(0, sp);
        const arg = sp === -1 ? "" : text.slice(sp + 1).trim();
        if (runCommand(cmd, arg)) return;      // client-handled?
        if (extCommands.some((c) => c.value === cmd)) { // extension command
            postThread("/command", { name: cmd.slice(1), args: arg });
            return;
        }
    }
    postThread("/prompt", { text, images: images.length ? images : undefined });
}

4Autocomplete state machine

A single popup ($ac) drives five modes via the module-level acMode. updateAc() (line 1532) runs on every keystroke and picks the mode from the composer text + caret; acceptAc(run) (line 1593) applies the highlighted item differently per mode.

Mode Trigger Accept behaviour
command leading /, no space run or insert full command
dir /new <dir> Enter creates thread · Tab drills in (/)
file @token at caret splice @path mid-line, keep editing
bash Tab in a ! command splice bare path, keep editing
ext extension provider match splice value; Enter may run

app.ts:1532 — updateAc (mode selection)

function updateAc() {
    const v = $prompt.value;
    const caret = $prompt.selectionStart ?? v.length;

    if (v.startsWith("/") && !/\s/.test(v)) {          // command palette
        acMode = "command";
        acItems = fuzzyFilter([...COMMANDS, ...extCommands], v, (c) => c.label);
        acIndex = 0; renderAc(); return;
    }
    const dm = v.match(/^(\/new\s+)(.*)$/s);            // /new <dir>
    if (dm) { acMode = "dir"; acAtStart = dm[1].length; acAtEnd = v.length;
        showDirAc(dm[2]); return; }
    const tok = atTokenBeforeCaret(v, caret);           // @file
    if (tok) { acMode = "file"; acAtStart = tok.start; acAtEnd = tok.end;
        showFileAc(tok.query); return; }
    // bash mode is force-opened by Tab, kept live while shown …
    if (extAcEnabled && v.trim()) { showExtAc(v, caret); return; }
    closeAc();
}
Race guard: every async source (showFileAc, showDirAc, showExtAc, showBashFileAc) captures const myReq = ++acReq; and bails if myReq !== acReq after the await — so a slow /files fetch can never repaint over a newer keystroke's results.

5SSE stream & frame dispatch

Exactly one EventSource is live at a time, scoped by ?thread. Switching threads means closing and reopening it; the host replays the thread's transcript on connect, so no separate "load" call is needed.

app.ts:2320 — reopenStream

let es: EventSource | null = null;
function reopenStream() {
    if (es) es.close();
    es = new EventSource(streamUrl());  // "/events?thread=id"
    es.onopen  = () => { $status.textContent = "● connected"; $status.className = ""; };
    es.onerror = () => { $status.textContent = "○ disconnected"; };
    es.onmessage = onSseMessage;
}

onSseMessage (line 2333) is a switch (m.kind) that fans ~30 frame types out to render functions. Selected arms:

app.ts:2333 — onSseMessage (excerpt)

function onSseMessage(e: MessageEvent) {
    const m = JSON.parse(e.data);
    switch (m.kind) {
        case "config":  if (typeof m.cwd === "string") { cwd = m.cwd; updateTitle(); } break;
        case "theme":   for (const [k, v] of Object.entries(m.vars))
                        document.documentElement.style.setProperty(k, String(v)); break;
        case "user":    bubble("user", m.text, m.images); pushHistory(m.text);
                     assistantEl = null; break;
        case "delta":   if (!assistantEl) { assistantEl = bubble("assistant", ""); assistantRaw = ""; }
                     assistantRaw += m.text; scheduleAssistantRender(); break;
        case "thinking": if (m.status === "start" || m.status === "full") newThinking().apply(m);
                     else (lastThinking() ?? newThinking()).apply(m); break;
        case "tool":    applyToolFrame(m); break;
        case "working": setWorking(!!m.busy); break;
        case "queue":   queuedMessages = Array.isArray(m.items) ? m.items : []; renderQueue(); break;
        // … surfaces, footer, header, title, threads, thread_switched,
        //    transcript_reset, thinking_level, editor, bash, custom, system, error
    }
}

6Streaming assistant / bash / tools

Throttled markdown

delta frames arrive token-by-token. Rather than re-parse markdown per token, renders are coalesced to one paint per animation frame:

let assistantRenderPending = false;
function scheduleAssistantRender() {
    if (assistantRenderPending) return;
    assistantRenderPending = true;
    requestAnimationFrame(() => {
        assistantRenderPending = false;
        if (assistantEl) renderAssistant(assistantEl, assistantRaw);
    });
}

Tool cards keyed by call-id

applyToolFrame (line 560) looks up or creates one <pi-tool> per call id and feeds it the frame. Auto-follow only fires when the user is near the bottom (followBottom checks the stickToBottom flag).

app.ts:560 — applyToolFrame

function applyToolFrame(m: any) {
    let el = toolCard(m.id);                // querySelector pi-tool[call-id=…]
    if (!el) {
        clearEmpty();
        el = document.createElement("pi-tool") as PiTool;
        el.callId = m.id; el.setAttribute("call-id", m.id);
        if (toolsExpandedDefault) el.info.expanded = true;
        $transcript.appendChild(el);
    }
    el.apply(m, cwd);                     // element owns its own render
    followBottom();
}

7Extension node renderer

Extensions describe UI as a serializable tree. renderNode(node, surfaceId) handles the three interactive types itself (wiring their actions back to /action) and delegates everything static to the shared renderStaticNode, recursing back through renderNode for children so interactive nodes nested in a Box still work.

app.ts — renderNode (interactive arms)

switch (node.type) {
    case "Button": {
        const b = document.createElement("button");
        b.textContent = node.label ?? "button";
        b.onclick = () => postThread("/action", { surfaceId, action: node.action });
        return b;
    }
    case "Frame": {                 // sandboxed arbitrary HTML/CSS/JS
        const frame = document.createElement("pi-frame") as PiFrame;
        frame.surfaceId = surfaceId ?? "";
        frame.frameHtml = node.html ?? "";
        return frame;
    }
    default:                        // Box/Row/Text/Markdown/Image/Spacer …
        const el = renderStaticNode(node, (c) => renderNode(c, surfaceId));
        return el ?? /* [unknown node] */;
}
<pi-frame> iframes emit bubbling piframe-action / piframe-notify events, routed once at the document level: actions → postThread("/action", …), notifies → toast(). Surfaces fan out to four regions: renderDock (above/below editor), renderOverlays, renderStatus, and dialog.render.

8Keyboard precedence

Two layers. The composer's keyGuard (line 2187) claims keys before its own history/submit handling; returning true means "handled". A separate document-level Escape handler enforces a strict fall-through order.

app.ts:2187 — composer.keyGuard (excerpt)

composer.keyGuard = (e) => {
    if (e.key === "Tab" && e.shiftKey) {          // cycle reasoning level
        e.preventDefault(); postThread("/thinking-level", { op: "cycle" }); return true;
    }
    if (e.key === "ArrowUp" && e.altKey && queuedMessages.length) { // restore queue
        e.preventDefault(); restoreQueue({ abort: false }); return true;
    }
    if ($ac.classList.contains("show")) {          // AC nav owns arrows/Tab/Enter/Esc
        switch (e.key) { case "ArrowDown": … case "Enter":
            acceptAc(acMode !== "file" && acMode !== "bash"); return true; }
    }
    // Tab with no AC open → force-open bash file completion
    return false;  // let the composer handle history / Enter-submit
};

Global one-shot keys live in their own document listeners, using e.code (physical key) so Option/Alt remaps on macOS don't break the match:

  • Alt+T (e.code === "KeyT") → toggleThinking()
  • Alt+O (e.code === "KeyO") → setAllToolsExpanded(!toolsExpandedDefault) — session-wide, not per-card

9Thread lifecycle

gotoThread is the pivot: it updates activeThreadId, resets the file cache (a new thread may have a different cwd), pushes the URL, and re-points the stream.

app.ts — gotoThread

function gotoThread(id: string) {
    if (!id || id === activeThreadId) return;
    activeThreadId = id;
    fileCacheAt = 0;                                // refetch @-files for new cwd
    history.pushState({ thread: id }, "", "?thread=" + encodeURIComponent(id));
    reopenStream();                                // host replays the new transcript
    updateTitle();
}

Initial load either resumes the URL's thread or asks the host to mint a fresh one and canonicalizes the URL — deliberately not resuming the most recent session:

app.ts — first-load bootstrap (tail of file)

if (activeThreadId) {
    reopenStream();
} else {
    post("/threads", {})
        .then((r) => r.json())
        .then((d) => { if (d?.id) {
            activeThreadId = d.id;
            history.replaceState({ thread: d.id }, "", "?thread=" + encodeURIComponent(d.id));
        }})
        .finally(() => reopenStream());
}
// popstate (browser back/forward) re-reads urlThread() and reopens the stream.
Per-thread reset: the transcript_reset frame clears $transcript, re-pins the welcome banner, empties the steering queue, resets stickToBottom = true, and drops assistantEl / working state — so a fresh thread starts clean and pinned to the bottom.
