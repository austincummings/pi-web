# Extension Points: pi TUI vs. pi-web

A side-by-side map of what the **pi TUI** exposes to extensions versus what the
**pi-web** currently bridges to the browser. Tracks TODO #16.

Sources:

- pi TUI: `/opt/pi-coding-agent/docs/extensions.md` (ExtensionAPI, ExtensionContext, `ctx.ui`).
- pi-web: `src/host/piweb-host.ts` (host registry on `globalThis.__PIWEB__`),
  `src/sdk/piweb.ts` (the `piweb` shim), `src/web/tools.ts` (client renderers),
  `src/web/app.ts` (in-iframe `window.piweb`).

> **Framing.** pi-web is **not** a second extension runtime. Extensions still run
> in-process under pi; pi-web only adds a _UI bridge_ so an extension can paint
> the browser web UI instead of (or alongside) the terminal. So the events /
> tools / session-control layers below are **inherited from pi unchanged** — they
> are not re-exposed by pi-web — while only the **`ctx.ui` surface** has a web
> counterpart. The matrix therefore focuses on the UI layer.

## 2. UI extension points — the matrix

pi's `ctx.ui` (and its message/tool render hooks) vs. pi-web's `piweb` surface.

Legend: ✅ implemented · 🟡 partial / proposed · ❌ missing · ➖ N/A in a browser

| #   | pi TUI (`ctx.ui.*` / render hooks)                            | pi-web counterpart                                                                |                    Status                    | Notes                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --- | ------------------------------------------------------------- | --------------------------------------------------------------------------------- | :------------------------------------------: | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 1   | `notify(msg, level)`                                          | `piweb.notify(msg, type)`                                                         |                      ✅                      | Toasts; same name, info/warning/error.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| 2   | `setStatus(key, text)`                                        | `piweb.setStatus(key, text)`                                                      |                      ✅                      | Exact pi-tui signature; keyed footer segments, `undefined`/empty clears.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                               |
| 3   | `setWidget(key, lines, {placement: aboveEditor                | belowEditor})`                                                                    | `piweb.setWidget(key, content, {placement})` | ✅                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                     | Same name + the two pi-tui slots (`aboveEditor`/`belowEditor`); `content` accepts `string[]` (drop-in for plain pi) or a serializable WidgetDef. Removal is `setWidget(key, undefined)`; the legacy `dock`/`removeWidget`/`clear` remain host-internal, off the public parity surface. |
| 4   | `select / confirm / input / editor` dialogs                   | `piweb.select/confirm/input/editor(...)` (blocking) + `piweb.overlay(id, def)`    |                      ✅                      | Blocking dialogs now `await` a browser answer: the open dialog rides the surfaces snapshot (replayed on reconnect) and `POST /ui-response` settles the promise. Same pi-tui signatures + `{signal, timeout}`. Declarative `overlay` cards remain for non-blocking modals.                                                                                                                                                                                                                                                                                                                                                                              |
| 5   | Custom tool rendering (`renderCall` / `renderResult`)         | `registerToolRenderer(name, fn)` (`src/web/tools.ts`)                             |                      ✅                      | Returns a DOM node or `null` to fall back; web equivalent of the TUI render hooks (TODO #4).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| 6   | `pi.registerMessageRenderer(customType, renderer)`            | `piweb.registerMessageRenderer(customType, (message, options, theme) => tree)`    |                      ✅                      | Matches pi-tui's `pi.registerMessageRenderer` (an ExtensionAPI method, not `ctx.ui`): the renderer takes `(message, options, theme)` (the pi-tui `MessageRenderer` shape) but returns a serializable node tree (`Box/Container/Row/Text/Markdown/Input/Button/Divider/Frame`; `Box`/`Container`/`Text`/`Markdown`/`Input` mirror the pi-tui components) instead of a live `Component`. Custom messages (`pi.sendMessage({ customType })`) render via the tree on live `message_end` + replay; unrendered types fall back to markdown. Highlighting fenced code inside `Markdown`/the transcript (the theme `--syn-*` palette) is the rest of TODO #19. |
| 7   | `custom(factory, options?)`                                   | `piweb.custom((theme, done) => surfaceDef, options?)`                             |                      ✅                      | Matches pi-tui `ctx.ui.custom` signature/return: the factory gets `(theme, done)` and returns a _serializable_ surface def (`{ render, actions?, initialState?, title? }`) instead of a live `Component`; it mounts as an overlay immediately and the returned `Promise` settles when the extension calls `done(result)`. `options` mirror pi-tui (`{ overlay?, overlayOptions?, onHandle? }`); `onHandle` yields `{ close, requestRender }`. Actions dispatch via `host.dispatch`. The former `overlay`/`openOverlay`/`closeOverlay`/`removeOverlay`/`remove` verbs are now internal plumbing behind `custom()`.                                      |
| 8   | `setFooter(factory)`                                          | `piweb.setFooter((data, theme) => tree)` + `piweb.refreshFooter()`                |                      ✅                      | Factory gets live `FooterData` (cwd/session/model/thinking/tokens/context/**`gitBranch`**/`availableModels`/`statuses`) + theme vars; returned node tree replaces `#contextbar`. `FooterData.gitBranch` is host-native + reactive (a `.git` watcher, pi-tui `getGitBranch`/`onBranchChange` parity). See `.pi/extensions/status-footer`.                                                                                                                                                                                                                                                                                                               |
| 8b  | `setHeader(factory)`                                          | `piweb.setHeader((data, theme) => tree)` + `piweb.refreshHeader()`                |                      ✅                      | Serializable node tree rendered as a custom header above the transcript (`#customheader`); `custom: null` restores the built-in header. Same `FooterData` + theme vars as `setFooter`.                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| 9   | `setWorkingMessage / setWorkingVisible / setWorkingIndicator` | `piweb.setWorkingMessage/setWorkingVisible/setWorkingIndicator(...)`              |                      ✅                      | Matches pi-tui: overrides the streaming "working" row. The host broadcasts a `working_config` frame (replayed on connect); the client applies message text, row visibility, and the spinner frames/`intervalMs` (omit to restore the braille default; `frames: []` hides the indicator; custom frames render verbatim). Per-thread; portable no-op under plain terminal pi.                                                                                                                                                                                                                                                                            |
| 10  | `setTitle(text)`                                              | `piweb.setTitle(text)`                                                            |                      ✅                      | Host emits a `title` SSE frame; client sets `document.title`. Default (no override) tracks the session as `π web - <thread name> - <cwd>` (mirrors the TUI's `π - <session> - <cwd>`, dropping segments that are absent); empty restores it. `ctx.ui` wiring tracked in #22.                                                                                                                                                                                                                                                                                                                                                                           |
| 10c | `setHiddenThinkingLabel(label?)`                              | `piweb.setHiddenThinkingLabel(label?)`                                            |                      ✅                      | Host-global label shown in place of a collapsed thinking block; broadcast as a `thinking_label` frame + replayed on connect. Empty/undefined restores pi's default ("Thinking...").                                                                                                                                                                                                                                                                                                                                                                                                                                                                    |
| 11  | `setEditorText / getEditorText / pasteToEditor`               | `piweb.setEditorText/getEditorText/pasteToEditor`                                 |                      ✅                      | Host writes the composer via an `editor` frame (`set` replaces, `paste` splices at the caret); the client debounce-echoes its text back over `POST /editor-text` so `getEditorText()` reads the live value (per-thread shadow, replayed on connect).                                                                                                                                                                                                                                                                                                                                                                                                   |
| 12  | `addAutocompleteProvider(factory)`                            | `piweb.addAutocompleteProvider((current) => (ctx) => items \| {start,end,items})` |                      ✅                      | Matches pi-tui's factory signature (`(current) => provider`): the factory wraps the current composed provider so a provider can add completions or defer to `current`. Each provider gets the composer `{ text, caret, cwd }` in-process and returns completion items (+ optional replace span); the browser queries the composed provider over `POST /autocomplete` as you type and splices the accepted value (Tab inserts, Enter runs). Portable: no-ops under plain terminal pi.                                                                                                                                                                   |
| 13  | `getToolsExpanded / setToolsExpanded`                         | `piweb.getToolsExpanded/setToolsExpanded` (+ per-card alt+o)                      |                      ✅                      | `setToolsExpanded(bool)` broadcasts a `tools_expanded` frame that expands/collapses every `<pi-tool>` card and sets the default for new ones; `getToolsExpanded()` returns the per-thread flag (reset on reload, replayed on connect). The alt+o/click per-card affordance remains.                                                                                                                                                                                                                                                                                                                                                                    |
| 14  | `setEditorComponent / getEditorComponent`                     | —                                                                                 |                      ➖                      | TUI Component swap (vim/emacs editors); not meaningful in the DOM composer.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| 15  | `getAllThemes / getTheme / setTheme / theme.fg(...)`          | `piweb.getAllThemes/getTheme/setTheme` + `piweb.theme`                            |                      ✅                      | `getAllThemes()` lists loadable themes (agent `themes/*.json`); `setTheme(name)` recomputes the web CSS-var palette (`loadPiTheme(name)`), persists it to `settings.json`, and rebroadcasts the `theme` frame to every viewer; `piweb.theme`/`getTheme(name)` expose the web-palette `Theme` shim (`theme.fg(...)`). The shim palette is fixed to the web vars (per-theme color extraction needs pi internals).                                                                                                                                                                                                                                        |
| 16  | In-frame host calls                                           | `window.piweb.action(name, payload)` + `window.piweb.notify`                      |                      ✅                      | Web-only: lets iframe widget content call back into the host (no TUI analogue).                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                        |
| —   | `ctx.hasUI` / `ctx.mode` guards                               | `piweb.present` flag + `piweb.mode === "web"`; no-op `stub` when no host          |                      ✅                      | Portable: calls degrade to no-ops under plain terminal pi. `piweb.mode` is the `ctx.mode` analog (the SDK `ExtensionMode` union has no `"web"`), `undefined` under the stub.                                                                                                                                                                                                                                                                                                                                                                                                                                                                           |

## 3. pi-web surface inventory (for reference)

Everything currently on `globalThis.__PIWEB__` (`src/host/piweb-host.ts`):

| Method                                                               | Purpose                                                                             | TUI analogue                          |
| -------------------------------------------------------------------- | ----------------------------------------------------------------------------------- | ------------------------------------- |
| `setWidget(key, content, {placement})`                               | Mount/replace a widget (aboveEditor/belowEditor)                                    | `setWidget(aboveEditor\|belowEditor)` |
| `removeWidget(key)` / `dock(id, def)` / `clear()` _(internal)_       | Convenience removal / legacy mount / drop-all - off the public surface              | `setWidget(key, undefined)`           |
| `custom(factory, options?)`                                          | Show a custom component (overlay); resolves on `done(result)`                       | `ctx.ui.custom`                       |
| `overlay/openOverlay/closeOverlay/removeOverlay/remove` _(internal)_ | Overlay surface primitives behind `custom()`                                        | — (pi-tui uses `custom`)              |
| `select/confirm/input/editor(...)`                                   | Blocking dialog; resolves on the browser's `/ui-response`                           | `ui.select/confirm/input/editor`      |
| `resolveUiRequest(id, value)` _(internal)_                           | Settle an open dialog with the browser's answer                                     | —                                     |
| `notify(msg, type)`                                                  | Transient toast                                                                     | `ui.notify`                           |
| `setStatus(key, text, opts?)`                                        | Keyed footer status segment                                                         | `ui.setStatus`                        |
| `registerMessageRenderer(customType, renderer)`                      | Custom transcript-message renderer (serializable tree; `(message, options, theme)`) | `pi.registerMessageRenderer`          |
| `addAutocompleteProvider(factory)`                                   | Composer autocomplete provider factory (queried via `/autocomplete`)                | `ctx.ui.addAutocompleteProvider`      |
| `hasAutocomplete()` / `autocomplete(ctx)` _(internal)_               | Provider-presence gate / run providers for a snapshot                               | —                                     |
| `renderMessage(customType, message, opts)` _(internal)_              | Invoke a registered renderer -> tree or null                                        | -                                     |
| `present`                                                            | Host-presence flag (false in the no-op stub)                                        | `ctx.hasUI`                           |
| `dispatch(id, action, payload)` _(internal)_                         | Run a surface action in-process, rebroadcast                                        | —                                     |
| `snapshot()` _(internal)_                                            | Serialize docks/overlays/status for SSE                                             | —                                     |

Client-side (`src/web/`): `registerToolRenderer(name, fn)` (tools.ts) and
in-iframe `window.piweb.{action,notify}` (app.ts).

## 4. Gap summary

- **Strong parity:** `notify`, `setStatus`, `setTitle`, custom tool rendering, blocking dialogs, host-presence guard.
- **Landed:** `setWidget` rename + widened placement (TODO #20); blocking `select/confirm/input/editor` dialogs (TODO #22); extension composer autocomplete (`addAutocompleteProvider`, TODO #12); `custom(factory, options?)` matching pi-tui's `ctx.ui.custom` (overlay verbs demoted to internal plumbing).
- **Proposed / in flight:** auto-routing fenced code blocks through a `"code"` renderer (rest of TODO #19).
- **Landed (TODO #22, this pass):** editor-text bridge (`setEditorText`/`getEditorText`/`pasteToEditor`), theme API (`getAllThemes`/`getTheme`/`setTheme`/`theme`), programmatic tool expansion (`getToolsExpanded`/`setToolsExpanded`), and the `ctx.mode` analog (`piweb.mode === "web"`). No outstanding UI-bridge gaps except the N/A `setEditorComponent`/`getEditorComponent`.
- **Landed:** `setFooter(factory)` + `refreshFooter()` and `setHeader(factory)` + `refreshHeader()` — full footer/header replacement, the pi-web analog of pi-tui's `ctx.ui.setFooter`/`setHeader` (TODO #22). `FooterData` is a faithful superset of pi-tui's `ReadonlyFooterDataProvider` (host-native reactive `gitBranch`, `availableModels`, inline `statuses`); factories also receive the active theme vars for explicit `Text` `color` (the `theme.fg(...)` analog).
- **Also landed:** `custom(factory, options?)` (pi-tui `ctx.ui.custom`); working-indicator overrides (`setWorkingMessage/Visible/Indicator`).
- **N/A in a browser:** `setEditorComponent` (TUI Component swap).
